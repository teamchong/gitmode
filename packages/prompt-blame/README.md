# `@gitmode/prompt-blame`

R&D package for capturing and querying prompt provenance on git commits. Designed to extend [Cloudflare Artifacts](https://developers.cloudflare.com/artifacts/) (or any Git remote) with agent-aware metadata.

> **Status:** prototype. Schema and API are not stable.

## What it does

For every commit produced by an AI agent, capture:

- **Which prompt** generated it (prompt ID, optional prompt text)
- **Which model** (e.g., `claude-opus-4-7`)
- **Which agent** (e.g., `claude-code`, `cursor`, `copilot`)
- **Which session/conversation** it belongs to
- **Whether a human edited it** before the commit landed

Then expose:

- `POST /metadata` — capture metadata for a `(repo, commit_sha)`
- `GET /metadata/:repo/:sha` — return what was captured for one commit
- `GET /blame/:repo/:sha?path=foo.ts` — *(future)* per-line provenance, joining git blame with metadata

## Design decisions

### D1, not Durable Object SQLite

Prompt-blame is read-heavy, write-modest, and **queries span repos** (e.g., "which sessions touched this file across the whole org"). DO SQLite is per-DO and would force one-DO-per-repo with awkward cross-repo queries. D1 is multi-tenant SQL across all Workers — the right shape.

Trade-off: D1 has weaker write consistency than DO SQLite. Acceptable for metadata sidecar — losing the last few writes during a regional failover is recoverable (the agent can resend).

### Commit-level only (line-level deferred)

Per-line provenance ("lines 1–15 generated by prompt X, line 7 human-edited") is the eventual goal but the costs are real:

- Storage scales with line count × commits, not commits
- Squash/rebase invalidates per-line records unless we track origin across rewrites
- Computing "human-edited" requires diffing the agent's proposed output against the actual commit — expensive

For Phase 2 we store **per-commit** metadata only. Per-line provenance arrives in a later phase, once we have a real workload to size against.

### No prompt text by default

`prompt_text` is **opt-in**. Prompts routinely contain:

- API keys mistakenly pasted in
- Internal codebase snippets
- Customer-identifiable data
- Confidential design discussions

Default is `prompt_id` only (a hash or UUID). When the user opts in, prompt text goes to **R2 keyed by prompt_id hash**, not D1 — D1 isn't sized for blob storage.

### Repo identifier shape

We use the **canonical Artifacts repo URL** (or any Git remote URL) as `repo_id`, normalized to lowercase. Examples:

- `https://x.artifacts.cloudflare.net/git/repo-13194.git`
- `https://github.com/user/project.git`
- `git@gitea.example.com:user/project.git` *(SSH form normalized to https where possible)*

Stable, human-readable, no separate registration step. If we need shorter keys later, we can hash + alias.

### Authentication (Phase 2 = none, future = token-bound)

The prototype Worker accepts unauthenticated writes. **This is unsafe for any real deployment** and is documented loudly.

The shipping plan:

1. Phase 2: no auth, in-memory rate limit, document the gap
2. Phase 4: caller passes the Artifacts repo token → Worker validates against Artifacts API → metadata writes scoped to repos the token can write

This means we depend on Artifacts' auth surface rather than building our own.

## Composition with `../timeline`

`../timeline` is the local-side prompt capture (Claude Code hook → snapshot via git notes). Composition:

```
Claude Code edit
  → timeline hook captures snapshot + parses session data
  → POST /metadata { repo, commit_sha, prompt_id, model, agent, session_id }
  → D1 insert
  ← 200 OK
```

Reverse path (query):

```
GET /metadata/:repo/:sha
  → D1 lookup
  ← { prompt_id, model, agent, session_id, human_edited, ... }
```

## Usage

### Run locally

```bash
# Apply migrations to a local D1 database
pnpm db:migrate:local

# Start the Worker on http://localhost:8787
pnpm dev

# Or pick a port
pnpm wrangler dev --port 8788
```

### CLI client (`bin/prompt-blame.mjs`)

```bash
# Post metadata for the current HEAD commit
node bin/prompt-blame.mjs post \
  --agent=claude-code \
  --model=claude-opus-4-7 \
  --session-id=abc123 \
  --human-edited=false

# Read it back
node bin/prompt-blame.mjs get
# {"repo_id":"...","commit_sha":"...","agent":"claude-code",...}

# Override target Worker
node bin/prompt-blame.mjs post --worker=https://prompt-blame.example.workers.dev ...

# Or via env
export PROMPT_BLAME_URL=https://prompt-blame.example.workers.dev
node bin/prompt-blame.mjs post --agent=claude-code ...
```

The CLI auto-detects `repo_id` from `git config --get remote.origin.url` (with normalization — SSH and HTTPS forms collapse to the same canonical URL) and `commit_sha` from `git rev-parse HEAD`.

### Direct HTTP

```bash
curl -X POST http://localhost:8787/metadata \
  -H "content-type: application/json" \
  -d '{"repo_id":"https://github.com/user/repo.git","commit_sha":"<40-hex>","agent":"claude-code"}'

curl "http://localhost:8787/metadata?repo=https%3A%2F%2Fgithub.com%2Fuser%2Frepo.git&sha=<40-hex>"
```

## What this is not

- Not a rewrite of git blame — we store the metadata; clients run `git blame` and join.
- Not a place to store full conversation history — that goes in agent-specific systems (Claude Code session files, Cursor history, etc.). We store *references*.
- Not a code review tool. Querying provenance is a primitive; review tooling is a layer above.
